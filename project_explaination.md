## 1. State Schema: The Shared Clipboard

In LangGraph, the **State** acts as a shared memory (or clipboard) passed from node to node. Each node reads what it needs, performs work, and returns state updates.

Here is our state schema defined in `graph_builder.py`:

```python
class my_state(TypedDict):
    query: str
    decision: NotRequired[str]
    retrieval_results: Annotated[list[dict], operator.add]
    final_answer: NotRequired[dict]
    feedback: NotRequired[str]
    iteration_count: NotRequired[int]
    review_status: NotRequired[str]
```

### Key Breakdown & Technical Terms:

- **`query`** (`str`): The initial user question received from the frontend/API client.
- **`decision`** (`str`): Updated by the **Supervisor Node** (Query Router). Takes one of three routing values: `"semantic"`, `"internet"`, or `"hybrid"`.
- **`retrieval_results`** (`Annotated[list[dict], operator.add]`): 
  - **Technical Concept: Reducer / Accumulator (`operator.add`)**: Instead of overwriting existing data, `operator.add` appends new retrieval dictionaries into a list.
  - When `"hybrid"` decision triggers **parallel execution (Map-Reduce)**, both `semantic` and `internet` nodes fetch data simultaneously and append their contexts into this single list without overwriting each other.
- **`final_answer`** (`dict`): Updated by the **Writer Node** (`generation/generator.py`). Holds the structured response synthesized from retrieved contexts.
- **`feedback`** (`str`): Updated by the **Reviewer Node** (`generation/reviewer.py`). Contains critique and instructions if the answer fails quality checks.
- **`iteration_count`** (`int`): Tracked by the **Writer Node** to count loop cycles. Serves as a **Loop Guard** (max 5 iterations) to prevent infinite reflection loops.
- **`review_status`** (`str`): Updated by the **Reviewer Node** with either `"pass"` or `"fail"`. Used by the conditional edge to route the flow.

## 2. Graph Nodes & Data Movement

Instead of diving deep into heavy code walkthroughs for every node, here is a clean overview of what each part reads, updates, and passes along:

```
[START]
   │
   ▼
┌──────────────────┐  Reads: query
│ Supervisor Node  │ ─────────────► Updates: decision
└──────────────────┘
   │
   ├── (if "semantic") ────────────────► [Semantic Node]  ──► Appends to: retrieval_results
   ├── (if "internet") ────────────────► [Internet Node]  ──► Appends to: retrieval_results
   └── (if "hybrid" Map-Reduce) ───────► [Both Parallel] ──► Appends to: retrieval_results
                                              │
   ┌──────────────────────────────────────────┘
   ▼
┌──────────────────┐  Reads: query, retrieval_results, feedback (if loop)
│   Writer Node    │ ─────────────► Updates: final_answer, iteration_count (+1)
└──────────────────┘
   │
   ▼
┌──────────────────┐  Reads: query, final_answer, retrieval_results
│  Reviewer Node   │ ─────────────► Updates: review_status ("pass"/"fail"), feedback
└──────────────────┘
   │
   ├── (if review_status == "fail" AND iteration_count < 5) ──► Loop back to [Writer Node] (Self-Reflection)
   └── (if review_status == "pass" OR iteration_count >= 5) ──► [END]
```

### Node Summary:
1. **Supervisor Node**: Evaluates query intent and sets `decision`.
2. **Semantic / Internet Nodes**: Perform vector search (local RAG) or web search, updating `retrieval_results`.
3. **Writer Node**: Blends context with the user query (incorporating `feedback` if re-writing) and updates `final_answer`.
4. **Reviewer Node**: Checks answer accuracy against retrieved context. If flawed, it sets `review_status = "fail"` and writes `feedback`.

## 3. End-to-End Flow (Frontend ↔ Backend ↔ Graph)

Here is how data moves across the full stack when a user asks a question:

1. **Frontend (Streamlit UI - `app.py`)**:
   - User enters a prompt in the UI input box.
   - Streamlit makes an HTTP GET request to `http://localhost:8000/stream?query=...` with `stream=True`.

2. **Backend API (FastAPI - `main.py`)**:
   - Accepts the query and executes the graph asynchronously using `graph.astream_events(...)`.
   - Monitors streaming events specifically from the **Writer Node** (`on_chat_model_stream`).

3. **LangGraph State Machine (`graph_builder.py`)**:
   - Initializes `my_state` with `{"query": query}`.
   - Executes nodes sequentially/parallely according to supervisor decision and reviewer evaluations.
   - As tokens are generated by the LLM in the Writer node, FastAPI yields them real-time over `StreamingResponse` back to Streamlit, which renders them typewriter-style.
